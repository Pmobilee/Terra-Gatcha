# Visuals & Art Direction -- 50 Questions for Roadmap Expansion

> These questions explore pixel art style, animation systems, biome design,
> mining feedback, and visual polish. Answer them to refine the visual roadmap.

---

## A. Autotiling, Terrain Rendering & Biome Identity (Questions 1-10)

### Q1: 4-bit vs 8-bit bitmask -- which autotile complexity level do you want at launch?
**Context**: Phase 7.1 calls for "4-bit or 8-bit bitmask autotiling." 4-bit uses 16 tile variants per block type (cardinal neighbors only). 8-bit uses 47 variants via Wang tile reduction (including diagonals). With 5 base block types across 25 biomes, 4-bit means 2,000 tile sprites; 8-bit means 5,875. The current `drawBlockPattern()` in `MineScene.ts` renders one flat sprite per cell with zero neighbor awareness.
**My Advice**: Start with 4-bit (16-variant) autotiling for all 25 biomes at launch. The visual leap from "no autotiling" to 4-bit is roughly 90% of the total quality gain. Full 8-bit is the Terraria gold standard but the sprite production cost is nearly 3x higher. A pragmatic path: ship 4-bit everywhere, then upgrade your 5 most-visited biomes (Sedimentary, Topsoil, Volcanic, Crystalline, Fungal Forest) to 8-bit post-launch based on player time-spent data.

### Q2: How should autotiling handle cross-material transitions -- dirt meeting stone, stone meeting hard rock?
**Context**: DD-V2-008 and DD-V2-009 both mandate "smooth transitions" between different block types. The current renderer draws each block independently with no edge blending. In Terraria, same-type blocks autotile among themselves, but cross-type transitions use a separate marching-squares edge overlay system.
**My Advice**: Implement a two-layer approach: (1) each block renders its autotiled base sprite using neighbor-of-same-type bitmask, then (2) a semi-transparent edge mask overlay is drawn showing the adjacent material "creeping in" from each neighbor of a different type. You only need 4-8 edge sprites per material (top, bottom, left, right, plus optionally 4 corners), which scales linearly rather than quadratically with material count. This is how Starbound handles cross-biome blending.

### Q3: Where does the autotile index get computed -- per frame or on state change?
**Context**: `MineScene.update()` calls `redrawAll()` every single frame, which clears and redraws the entire visible grid. Adding a neighbor lookup per tile per frame for autotiling would multiply the per-frame cost significantly, especially on mobile where the grid could be 40x40 at deeper layers (Phase 8.2).
**My Advice**: Compute and cache the autotile index in each `MineCell` whenever that cell or any of its neighbors changes state (mined, revealed, or collapsed). Store it as a `tileVariant: number` field on `MineCell`. In `drawTiles()`, simply look up the cached variant. Additionally, move from full-grid-every-frame redraws to a dirty-rect system where only changed tiles and their 8 neighbors re-render. This drops per-frame draw from O(visible_grid) to O(changed_tiles), which is essential for 40x40 grids on mobile.

### Q4: How do you handle autotiling at biome boundaries within a single layer?
**Context**: `generateBiomeSequence()` in `biomes.ts` assigns one biome per layer, so currently there are no intra-layer biome boundaries. But Phase 9 mentions biome-specific structural generation (Volcanic lava rivers, Fungal organic tunnels), which implies biome features could appear within a layer that is primarily another biome. If two biome tile styles meet, the autotiling logic needs a plan.
**My Advice**: Create a "neutral transition" tileset with desaturated earth tones that can bridge any two biome palettes. Place it in a 2-3 tile transition zone between biome regions. This avoids the combinatorial explosion of generating unique transition tiles for every biome pair (25 x 24 = 600 pairs). Starbound uses exactly this approach. An alternative: define biome sub-regions within a layer but keep all blocks using the same autotile set, varying only the palette tint -- simpler but less visually distinct.

### Q5: Each of the 25 biomes needs unique tile sprites that are "fundamentally different, not just color-shifted" (DD-V2-014). How different is different enough?
**Context**: The current 3 biomes (Sedimentary, Volcanic, Crystalline) use the same 5 base tile sprites (`tile_dirt`, `tile_soft_rock`, `tile_stone`, `tile_hard_rock`, `tile_unbreakable`) with color tint overlays applied in `drawTiles()`. The design doc explicitly rejects this approach. But generating 5 completely unique tile sets for 25 biomes is 125+ base sprites before autotiling.
**My Advice**: Define three tiers of biome visual uniqueness. Tier 1 (5-6 "hero" biomes like Fungal Forest, Data Ruins, Ancient City, Living Cave, Coral Reef): fully unique sprites with distinctive silhouettes and textures that are immediately recognizable at thumbnail scale. Tier 2 (10-12 biomes): unique textures but structurally similar shapes, differentiated primarily by material grain and color family. Tier 3 (7-8 biomes): shared structural sprites with heavy palette variation and 2-3 unique accent tiles per biome. This gives you the "not just color-shifted" mandate while keeping production manageable.

### Q6: How do you prevent palette fatigue across 25 biomes -- can you assign dominant hue, secondary accent, and signature motif for each right now?
**Context**: Several biomes overlap in color space: Volcanic and Magma Core are both red-orange; Crystalline and Crystal Caverns are both blue-purple; Geode Cathedral overlaps both. Salt Flats and Frozen/Permafrost are both white-blue. The `Biome` interface in `biomes.ts` only defines `bgColor`, `fogTint`, and `blockColorOverrides` -- there is no formal palette specification.
**My Advice**: Before generating a single new biome sprite, create a 25-biome color matrix plotted on a hue wheel. Ensure no two biomes that could appear in adjacent layers share the same dominant hue. Differentiate "sibling biomes" (Volcanic vs Magma Core) through structural silhouette rather than color: Volcanic uses horizontal lava river patterns; Magma Core uses vertical column formations with drip stalactites. When two biomes share a hue, their shape language must be opposite. Add `palette: { dominant: number, accent: number, highlight: number }` to the `Biome` interface as a formal contract.

### Q7: The Topsoil biome is the tutorial layer. What visual language communicates "safe, explore freely" vs the deep biomes that say "danger ahead"?
**Context**: Phase 14.4 describes a handcrafted tutorial mine that should feel warm and inviting. The deepest biomes (Primordial, Magma Core, Data Ruins) should feel alien and threatening. Currently all 3 biomes use a uniformly dark underground palette with only `bgColor` (0x111118 to 0x1a0a0a) varying slightly.
**My Advice**: Establish a depth-based visual gradient. Topsoil: warm earth tones (amber/ochre/sage green), organic rounded shapes (roots, worms, soft dirt clumps), high ambient brightness, no hazard particles. Mid-depth biomes: cooler temperatures (teal/gray/slate), more angular shapes, moderate brightness, scattered ambient particles. Deep biomes: cold palette (indigo/purple/void black), sharp jagged geometry, low base brightness with high-contrast glow effects, heavy ambient particles. This warm-to-cold, round-to-angular, bright-to-dark gradient is universally intuitive.

### Q8: How many per-biome animated tile types can the mobile GPU budget support?
**Context**: Phase 7.7 lists 10+ block types that need idle animations (crystal glinting, lava flowing, gas swirling, etc.). Phase 9 adds biome-specific ambient effects. The "Living Cave" biome has pulsing walls; "Fungal Forest" has bioluminescent mushrooms. These require animated tiles. The current renderer uses `Math.sin(this.time.now / N)` for overlay pulsing (e.g., lava at line 685 of MineScene.ts) but has no sprite-sheet animation system.
**My Advice**: Budget 3-4 animated tile types per biome, each with 4-6 frames in a sprite sheet. On mobile with ~144 visible tiles, if 20% are animated, that is ~29 animated tiles updating per frame. Use a global animation timer with staggered phase offsets per tile (seeded by grid position) so all crystals do not pulse in sync -- the offset creates visual richness without extra frames. Implement a sprite-sheet frame cycling system rather than per-pixel shader effects. Pre-calculate which tiles are animated during reveal, not per-frame.

### Q9: The current biome system applies a colored overlay in `drawTiles()` for Volcanic (0xb43c1e at 0.15 alpha) and Crystalline (0x5078c8 at 0.15 alpha). When you switch to unique per-biome tile sprites, do you keep or remove these tint overlays?
**Context**: Lines 664-681 of `MineScene.ts` apply a semi-transparent biome tint on top of geology sprites for Dirt, SoftRock, Stone, and HardRock. If the new biome tile sprites already incorporate the biome's color identity, the overlay becomes a double-tint that muddies the art.
**My Advice**: Remove the programmatic tint overlay once biome-specific tile sprites are in place. The tint was a pragmatic stopgap for making 3 biomes feel different using shared sprites. With dedicated art per biome, the sprites should carry the palette themselves. However, keep a very subtle tint (0.05-0.08 alpha) as a unifying wash that ties heterogeneous block types together within a biome -- this is the pixel art equivalent of color grading in film.

### Q10: Should special blocks (Quiz Gate, Relic Shrine, Descent Shaft, etc.) have biome-specific appearances, or remain visually consistent across all biomes for gameplay clarity?
**Context**: Phase 9.2 mentions "biome-specific special block appearances (quiz gates look different in each biome, etc.)." But DD-V2-026 makes quiz gates mandatory for progression -- players must recognize them instantly. Currently all special blocks use a single universal sprite.
**My Advice**: Keep a recognizable structural silhouette for all gameplay-critical blocks (Quiz Gate, Descent Shaft, Exit Ladder) that is consistent across biomes, but allow the color palette and decorative details to adapt. A Quiz Gate should always read as "glowing doorway with rune symbols" but the rune color, frame material, and glow hue can match the biome. For less critical special blocks (Quote Stone, Relic Shrine), full biome-specific redesigns are fine. The rule: if the player must find it to progress, its shape must be universal.

## B. Character Animation, Mining Feel & Pickaxe Satisfaction (Questions 11-20)

### Q11: How many frames per animation direction are you targeting for the miner sprite sheets?
**Context**: DD-V2-011 specifies 8+ sprite sheets (idle, walk x4, mine x3). Phase 7.2 suggests 4 frames idle, 6 frames per walk direction, 4 frames per mine direction. The current miner is a single static `miner_idle.png` sprite at `/src/assets/sprites/characters/`. At 32px tile size, the miner occupies one tile on screen.
**My Advice**: For walk cycles, target 6 frames minimum. At pixel art scale, 3 frames reads as jittery while 6 reads as fluid. For the mine/swing animation, target 5-6 frames with asymmetric timing: 2 anticipation frames (slow), 1 swing frame (fast), 1 impact frame (hold for 2-3 game ticks), 1-2 recovery frames. This "wind-up, snap, settle" rhythm is what makes Stardew Valley's tools feel satisfying. Total sprite count: 8 directions x 6 frames = 48 character frames, which is very manageable for ComfyUI generation as a single sheet.

### Q12: The game uses tap-to-mine on mobile. How do you reconcile a multi-frame swing animation with the instant feedback expectation of a tap?
**Context**: DD-V2-012B describes an "ultra-satisfying mining feedback loop" with wind-up, swing, impact, and recovery. But Stardew Valley's satisfying pickaxe swing uses held-click, not tap. On mobile, players expect to tap rapidly -- waiting for a full swing animation before the damage registers would feel laggy.
**My Advice**: Use a "compressed swing" for tap input. The impact frame plays immediately on tap (frame 1), damage is applied instantly, and the follow-through/recovery plays out over the next 3-4 frames purely as visual feedback. For multi-tap blocks, the recovery of one swing blends into the anticipation of the next, creating the "rhythmic flow" described in DD-V2-012B. The trick: separate the gameplay event (damage applied on tap) from the animation event (swing visuals play through). Never make the player wait for animation to complete before their next tap registers.

### Q13: Should you prototype mining feel with placeholder animations before commissioning final pixel art sprite sheets?
**Context**: Phase 7.8 says "dedicated prototyping, iteration, and playtesting" for pickaxe feel. Generating 48+ character frames via ComfyUI at 1024x1024 represents significant GPU time. If the timing feels wrong after seeing the final art, you would need to regenerate.
**My Advice**: Absolutely prototype first. Build the entire mining feel system using colored rectangles and circles in Phaser: a rectangle arm that rotates through swing arc keyframes, a circle impact burst, procedural screen shake, and placeholder particles. Iterate on timing curves, shake intensity, and particle density until the feel is right with these throwaway visuals. Only then generate the final pixel art sprite sheets to match the proven timing. This saves potentially dozens of hours of wasted ComfyUI generation.

### Q14: How do per-block-type impact effects differ -- what makes dirt "feel" different from crystal when you hit it?
**Context**: DD-V2-012B lists "dirt crumbles softly, stone clinks with sparks, crystal rings with prismatic shards, hard rock thuds with dust clouds." The current `spawnBreakParticles()` uses a single 4x4 white particle texture tinted to the block's color, with identical quantity and behavior for all block types.
**My Advice**: Define an impact profile per block material: Dirt -- 6 small brown particles with slow gravity, gentle screen nudge, soft thud. Stone -- 4 sharp gray particles with fast velocity, white spark flash, medium screen shake. Crystal -- 8 multi-colored particles with long lifespan and slow float, prismatic screen flash, chime-like audio cue. Hard Rock -- 3 heavy particles with fast gravity, dust cloud that lingers 0.5s, strong screen shake. Lava -- orange ember spray upward, heat shimmer distortion, sizzle. Each profile differs in particle count, velocity, gravity, lifespan, color spread, screen shake intensity, and flash color.

### Q15: Progressive hit satisfaction -- how does the 1st tap on a 5-hit block feel different from the 4th?
**Context**: DD-V2-012B requires "each successive hit on a multi-hit block feels increasingly impactful (wider cracks, more particles, screen shake intensifies slightly)." The current `drawCracks()` has two stages (34-66% and 67-99%) with procedural vector lines. Particle count and shake are currently identical per hit.
**My Advice**: Scale three parameters linearly with damage percentage. Particle count: 4 at 0% damage, 10 at 80% damage. Screen shake intensity: 1px at first hit, 3px at penultimate hit. Crack overlay opacity: 0.4 at first crack stage, 0.8 at near-break. Also add a subtle "bulge" effect on the later hits -- the block sprite scales to 1.03x for one frame then snaps back, simulating the block straining before it breaks. The break moment itself (final hit) should be a distinct peak: 15 particles, 5px shake, 150ms white flash, and a 50ms freeze-frame before the block disappears. The freeze-frame is the single most impactful juice trick.

### Q16: How do pickaxe tier visual upgrades manifest during mining? Is it a different swing animation, different particle color, or both?
**Context**: Phase 7.8 mentions "higher tier pickaxes have flashier swing animations, bigger impact effects, unique sound profiles." The current pickaxe tier system (`BALANCE.PICKAXE_TIERS`) defines 5 tiers from Stone (1 damage) to Plasma (8 damage) but has no visual representation beyond a HUD badge.
**My Advice**: At minimum, change the swing trail color per tier: Stone = gray, Iron = silver, Steel = blue-white, Diamond = cyan sparkle, Plasma = magenta energy trail. Higher tiers should also emit more particles on impact (6, 8, 10, 14, 20) and trigger slightly stronger screen shake. A full separate animation set per tier is ideal but expensive (48 frames x 5 tiers = 240 frames). A pragmatic middle ground: one animation set with a recolorable swing trail overlay. Generate the trail as a separate 4-frame sprite sheet and tint it per tier at runtime.

### Q17: Gear visibility on the miner -- should equipped pickaxe tier, scanner antenna, and full backpack be visually reflected on the character sprite during a dive?
**Context**: DD-V2-011 mentions "gear visual changes on miner sprite (pickaxe tier visible, scanner antenna, full backpack bulge)." Phase 7.2 includes this as a sub-task. But the miner is only 32px on screen -- there is very little pixel space for accessory detail.
**My Advice**: At 32px, subtle accessory changes will be invisible on mobile. Instead of modifying the base character sprite, show gear as small overlay icons floating near the miner: a tiny pickaxe icon to the right (colored by tier), a small antenna dot above (if scanner upgraded), and a subtle bulge or glow when the backpack is full. These overlays render at the sprite layer and follow the miner position. This approach avoids needing to generate 5x4x2 = 40 character variants for every equipment combination and actually reads better on small screens.

### Q18: How should the climactic block-break moment differ visually from an intermediate mining hit?
**Context**: DD-V2-012B describes the final hit as having "a distinct, climactic feel: block shatters, loot pops out, dust cloud, satisfying crunch." The current implementation instantly removes the block and emits 6 particles. There is a 150ms white flash overlay but no distinct break vs hit differentiation.
**My Advice**: The break moment needs three elements that intermediate hits do not have: (1) a 50ms freeze-frame where the entire game pauses (the most powerful juice technique -- it creates a sense of weight), (2) a radial particle burst with 15+ particles in all directions (vs the 4-6 directional particles of intermediate hits), (3) a block-shatter animation where the sprite splits into 4 quadrants that fly outward before fading. After the freeze-frame, loot items pop out with physics arcs. This sequence takes about 400ms total and transforms block breaking from "the block disappeared" to "I destroyed that."

### Q19: Loot pop physics -- what trajectory, bounce, and collection behavior feels best for minerals popping out of broken blocks?
**Context**: DD-V2-013 requires "minerals bounce out with physics arc, then get sucked into the backpack." The current system has no loot pop -- items are added to inventory immediately on block break with no visual representation.
**My Advice**: Use a parabolic arc with randomized initial velocity (angle 30-80 degrees upward, speed 60-120 px/sec) and 2-3 diminishing bounces on the ground plane. After the final bounce, hold the item stationary for 200ms, then vacuum-suck toward the player position with an ease-in-cubic curve over 300ms. Add a small "+1" text pop on collection and a brief inventory slot flash. Total sequence: 600-900ms from break to collection. For multi-item drops (mineral nodes with multiple drops), stagger the pop timing by 100ms per item so they fan out visually. The vacuum phase is the moment the reward emotionally registers.

### Q20: Haptic feedback via Capacitor -- what vibration patterns for mining hit, block break, and rare find?
**Context**: DD-V2-012B specifies "haptic feedback on mobile -- subtle vibration on hit, stronger on break." Capacitor provides `Haptics.impact()` with light/medium/heavy styles and `Haptics.vibrate()` for custom patterns. No haptics are currently implemented.
**My Advice**: Mining hit: `Haptics.impact({ style: 'light' })` -- 10ms soft tap. Block break: `Haptics.impact({ style: 'medium' })` -- 20ms firm tap. Rare item found: `Haptics.notification({ type: 'success' })` -- the iOS "success" triple-pulse pattern. Earthquake: `Haptics.vibrate({ duration: 300 })` -- sustained rumble. Hazard damage: `Haptics.notification({ type: 'warning' })`. Always provide a "disable haptics" toggle in settings. On Android, use `Haptics.impact({ style: 'heavy' })` for breaks since Android light haptics are often imperceptible.

## C. Crack Overlays, Particle Systems & Visual Juice (Questions 21-30)

### Q21: Sprite-based crack overlays vs the current procedural vector approach -- what are the damage stages, and should cracks look different per block type?
**Context**: DD-V2-010 mandates "sprite-based crack overlays with 3-4 damage stages." BUG-005 notes that `crack_small.png` and `crack_large.png` exist in `src/assets/sprites/tiles/` but are unused. The current `drawCracks()` in MineScene.ts (lines 429-490) draws procedural vector lines with seeded offsets -- 2-3 thin lines at 34-66% damage, 4-6 thick lines at 67-99%.
**My Advice**: Define 4 crack stages: (1) hairline -- barely visible stress marks, 1px lines; (2) fractured -- clear cracks radiating from center, 2px lines with small chip missing; (3) crumbling -- wide cracks with chunks separating, visible block interior; (4) near-break -- block visibly fragmented, pieces wobbling. Generate these as semi-transparent overlay sprites (32x32 and 256x256) with alpha channel. For block-type variation, you do not need unique crack sprites per material -- instead, tint the crack interior color to match the block type. Stone cracks show dark gray interior; crystal cracks show prismatic glint; dirt cracks show darker earth.

### Q22: What is the particle budget per device tier, and how do you scale effects gracefully on low-end mobile?
**Context**: Phase 7.4 lists 9 distinct particle effect types. The current `breakEmitter` uses a single 4x4 white texture with 6 particles per event. With ambient biome particles, mining particles, loot physics, and hazard warnings all active simultaneously, the total particle count could spike.
**My Advice**: Define three device performance tiers: Low (older phones) -- max 40 active particles, no ambient particles, reduced break particles (3 instead of 6). Medium (2020+ phones) -- max 80 particles, 10 ambient, full break particles. High (2022+ phones / desktop) -- max 150 particles, 20 ambient, enhanced break with longer lifespan. Detect tier via a framerate check during the first 5 seconds of gameplay. Store the tier in settings so players can also manually adjust. Use Phaser's particle pool with a hard `maxParticles` cap per tier.

### Q23: Biome-specific ambient particles -- how do you keep them from accumulating to lag-inducing levels during long exploration sessions?
**Context**: Phase 9.4 requires unique ambient particles per biome: embers in Volcanic, spores in Fungal, drips in Coral Reef, sparkles in Crystalline. These need to persist continuously while the player explores but the viewport moves constantly, revealing new areas.
**My Advice**: Use viewport-culled ambient emitters that only spawn particles within the visible camera rect plus a 2-tile margin. Particles that drift outside this rect are immediately recycled back to the pool. Set a hard cap of 15-20 ambient particles visible at any time. For biomes with multiple ambient types (Volcanic has embers AND heat shimmer), split the budget across types (10 embers + 5 shimmer = 15 total). Never let ambient particles persist outside the viewport -- this prevents "particle creep" during long sessions.

### Q24: Screen shake design -- what intensity tiers, and how do you handle players who find shake disorienting?
**Context**: Screen shake is called for in DD-V2-012B (mining impact), Phase 7.4 (bombs, earthquakes, cave-ins), and Phase 3.5 (unstable ground). The current implementation has no screen shake at all. The camera uses lerp-based smooth follow at 0.15 in `updateCameraTarget()`.
**My Advice**: Define three shake tiers: micro (1-2px displacement, 50ms, for block hits), medium (3-5px, 150ms, for block breaks and bombs), and heavy (6-10px, 300ms, for earthquakes and cave-ins). Use Perlin noise-based displacement rather than random jitter -- Perlin shake feels "physical" while random feels "glitchy." Critical rule: never shake the UI/HUD layer, only the game camera. Add a "screen shake intensity" slider in settings (0%, 25%, 50%, 100%) with 100% as default. This is an accessibility must-have for players with motion sensitivity.

### Q25: Gacha reveal animations -- how long per rarity tier, and what visual elements escalate between Common and Legendary?
**Context**: DD-V2-014 and Phase 7.5 define escalating reveal animations: "Common=shimmer, Rare=sparkle, Epic=burst, Legendary=full-screen lightshow+fanfare, Mythic=unique prismatic animation." The current `FactReveal.svelte` has basic CSS fade-in without rarity-dependent effects.
**My Advice**: Design from Legendary downward, not Common upward. Define maximum spectacle for Legendary (full-screen golden flash, 50 burst particles, 5px screen shake, radiating light rays, 3-second item hold, unique sound sting) then subtract elements for each lower tier: Epic drops the screen flash, Rare drops the light rays, Uncommon drops screen shake, Common is just a pop-out with 8 particles. Duration: Common 1.5s, Uncommon 2s, Rare 2.5s, Epic 3.5s, Legendary 4-5s. Always provide a "Skip" button that appears after the first 1 second. The contrast between tiers is what creates excitement, not the absolute spectacle of any one tier.

### Q26: The descent shaft falling animation -- how long, what does the player see, and how does it transition to the new biome?
**Context**: Phase 7.5 and 8.5 call for "falling animation when entering descent shaft (1-2 second dramatic fall)" with a "biome transition screen with biome name, visual preview, and layer number." Currently, descending to a new layer instantly restarts the MineScene with new parameters -- no animation at all.
**My Advice**: Design a 3-phase transition: (1) Fall (1s) -- camera zooms slightly out, miner drops into the shaft with a downward acceleration, shaft walls blur-scroll upward, particles trail behind. (2) Biome card (1.5s) -- full-screen card with biome name, layer number ("Layer 4: Fungal Forest"), a small biome icon/thumbnail, and a brief GAIA comment. Use the biome's dominant palette for the card background. (3) Arrive (0.5s) -- camera zooms back in on the new spawn position, brief dust-settle particle burst. Total: 3 seconds. This transforms a jarring scene restart into a cinematic moment that builds anticipation for the new biome.

### Q27: The current block idle animations use `Math.sin()` overlay pulsing. Should you migrate to sprite-sheet-based frame animations for a more pixel-art-authentic look?
**Context**: Lines 684-696 of MineScene.ts implement hazard idle animations as semi-transparent color overlays pulsing via sine wave (lava at 0.15+0.10*sin, gas at 0.10+0.06*sin, unstable ground at 0.05+0.03*sin). Phase 7.7 envisions "crystal glinting (2-3 frame loop), lava flowing (4-6 frames), gas swirling." These are qualitatively different from the current color-wash approach.
**My Advice**: Migrate to sprite-sheet frame cycling for blocks that need distinct animation shapes (lava flow, gas swirl, descent shaft vortex, quiz gate energy barrier). Keep the `Math.sin()` overlay approach for blocks that just need a subtle glow pulse (mineral nodes, relic shrines, crystal glint). The hybrid approach -- sprite animation for "active" blocks, overlay pulse for "ambient" blocks -- lets you focus the expensive sprite frames on the blocks that benefit most. Budget 4-6 frames per animated block type; use a shared global timer with per-tile phase offset.

### Q28: Mining progress visualization -- should there be a visible health bar per block, or do cracks alone communicate remaining hits?
**Context**: Phase 7.5 mentions "mining progress bar with satisfying fill animation." The current system shows procedural cracks at 34% and 67% damage, and the HUD has no per-block health indicator. On 5-hit hard rock, the player gets visual feedback only at hits 2 and 4.
**My Advice**: Do not add an explicit health bar floating above each block -- it would clutter the visual field and undermine the pixel art aesthetic. Instead, make cracks alone communicate progress by using 4 distinct crack stages (at 25%, 50%, 75%, and 90% damage) rather than the current 2 stages. Each new crack stage should trigger a micro screen shake and particle burst so the player feels the transition. If you still want numerical clarity, show a small "2/5" indicator briefly (0.5s fade) on the block after each hit, in a pixel font.

### Q29: Scanner sonar pulse -- when the player is near the descent shaft, what visual hint appears?
**Context**: Phase 7.5 includes "scanner sonar-style pulse visual when near descent shaft." The current scanner system shows rarity shimmer and hazard warnings on fog tiles (lines 549-616 of MineScene.ts) but has no proximity-based pulse effect.
**My Advice**: When the player is within 5-8 tiles of the descent shaft (distance scaling with scanner tier), emit a radial "ping" wave originating from the player every 2 seconds. The wave should be a thin expanding circle (1px line, 30% opacity) in the descent shaft's purple color (0x6633cc) that fades as it expands. When the wave passes over the shaft's tile, the shaft tile briefly glows brighter. This gives the player directional awareness ("it's somewhere to the right") without revealing the exact location. The effect only activates when the scanner tier is 2 or higher.

### Q30: How should the loot rarity be communicated at the moment of block break, before the full gacha reveal?
**Context**: When breaking an artifact node, the current system immediately opens a quiz overlay. There is no visual "preview" of the artifact rarity. DD-V2-013 envisions rarity-escalating break animations. The player should know something special happened at the moment of breaking a rare block.
**My Advice**: Use escalating break effects based on the loot rarity hidden inside the block: Common -- standard break particles, no special effect. Uncommon -- slightly larger particle burst with a green tint flash. Rare -- blue particle burst with a brief 1-tile glow ring. Epic -- purple burst with expanding glow ring and screen micro-shake. Legendary -- gold burst, 3-tile glow ring, heavy screen shake, 100ms freeze-frame. These effects fire at the break moment, before any overlay appears, giving the player that critical "wait, what was THAT?" reaction.

## D. Dome Hub, GAIA, and UI Visual Design (Questions 31-40)

### Q31: The dome is currently a single elliptical enclosure. The design doc calls for multi-floor vertical stacking. How do you envision the floor-to-floor transition -- side-view cutaway (Fallout Shelter style) or separate rooms per screen?
**Context**: DD-V2-016 mandates a "multi-floor glass hub system" with vertical scrolling. The current `DomeCanvas.svelte` renders a single 192x128 tile ellipse at 4px/tile with three interior levels (upper catwalk, main floor, basement). `DomeView.svelte` wraps it with a resource bar and slide-up panel.
**My Advice**: Use the side-view cutaway approach where all floors are visible simultaneously, stacked vertically. This gives players the emotionally satisfying feeling of seeing their entire base at once -- Fallout Shelter proved this is more engaging than separate room screens. Each floor is a self-contained horizontal slice sharing the dome's glass shell. The DomeCanvas would become taller (192 x (128 * floorCount)) and the viewport scrolls vertically. The current elliptical dome shape could transition to a cylindrical tower with glass panels at each floor level.

### Q32: How do you visually communicate "this floor is empty and waiting to be filled" vs "this floor is upgraded and complete"?
**Context**: DD-V2-019 says "all upgrades must be visually reflected in the hub." Phase 10.3 defines "visual upgrade tiers per floor (bare metal -> decorated -> lush/premium)." The current dome has all objects pre-placed with no empty/incomplete states.
**My Advice**: Empty floors should use a desaturated, low-contrast palette with exposed structural beams, flickering dim lights, and a single "construction" placeholder sprite. ## E. Fog of War, Color Management, Pipeline & Mobile Optimization (Questions 41-50)

### Q41: The fog of war silhouette ring renders at ~80% opacity dark overlay (0.80 + scanner * 0.05). On OLED mobile screens in bright rooms, is this visible enough?
**Context**: Lines 536-547 of MineScene.ts render Ring-1 (visibilityLevel=1) by drawing the actual tile sprite, then covering it with a dark overlay at (1.0 - borderBrightness) alpha, where borderBrightness is 0.80-0.90. The GAME_DESIGN.md defines Ring-1 as "~10% opacity ghost color," but the implementation shows 10-20% of the sprite through the dark overlay.
**My Advice**: Test on a real OLED device in daylight. In practice, the minimum perceptible color on a dark background on OLED in indoor light is ~18-22% brightness. Your current implementation showing 10-20% of the sprite is at the edge of visibility. Increase the base ring-1 brightness to 25-30% for mobile. Add a "fog contrast" accessibility slider in settings that scales from 15% (subtle) to 45% (high visibility). Players who cannot perceive the fog rings miss the entire exploration tension mechanic.

### Q42: Biome-specific fog colors (volcanic = orange glow, crystalline = blue shimmer) are mentioned in GAME_DESIGN.md. How do you implement colored fog without it looking like a garish overlay?
**Context**: The current `fogTint` field on each Biome (0x1a1a2e for Sedimentary, 0x2a1010 for Volcanic, 0x10102a for Crystalline) is used as a flat fill for hidden tiles. The design doc envisions colored shimmer in the fog zone, not just a colored hidden zone.
**My Advice**: Do not tint the fog overlay itself. Instead, add a subtle radial glow emanating from specific block types that "leaks" into the fog zone. Lava blocks in the fog emit a 1-tile warm orange radial glow; crystal nodes emit a cool blue glow. The fog stays neutral dark, but specific blocks signal their presence through it. This is more atmospheric than a biome-wide tint, doubles as a gameplay hint (sense lava before revealing it), and is already partially implemented via the scanner shimmer system on lines 549-592.

### Q43: Color palette management across 25 biomes -- do you need a formal palette system, or is ad hoc color picking sufficient?
**Context**: The current biome colors are hardcoded hex values in `biomes.ts` with no formal palette structure. As you scale to 25 biomes with unique sprites, the risk of color drift is high: a sprite generated on Monday might use slightly different blues than one generated on Friday, even with the same prompt.
**My Advice**: Create a formal `palette.ts` file that defines exactly 8-12 colors per biome (background, 3 block fill colors, accent, highlight, shadow, fog tint, particle color). Enforce these palettes during sprite generation by including the exact hex values in ComfyUI prompts or applying palette quantization as a post-processing step. Tools like Aseprite's palette remap or a custom ImageMagick script can force generated sprites into the canonical palette. Without this, your game will look like a collage of 25 different artists by the 15th biome.

### Q44: Texture atlas generation -- at 500+ sprites with autotile variants, will you pack them into atlases, or continue loading individual PNGs via Vite's import.meta.glob?
**Context**: The current `spriteManifest.ts` and `domeManifest.ts` use `import.meta.glob` to eagerly load every PNG individually. This works at ~65 sprites but at 500+ sprites, it means 500+ HTTP requests on first load (even with Vite bundling, each sprite is a separate decoded texture in GPU memory).
**My Advice**: Implement a texture atlas pipeline. Pack sprites into atlas sheets (2048x2048 max for mobile GPU compatibility) using a tool like TexturePacker or free-tex-packer during the build step. Generate a JSON sprite map alongside each atlas. In BootScene, load atlas sheets instead of individual files -- this reduces HTTP requests from 500+ to ~10 and improves GPU batch rendering performance. Phaser has native atlas support via `this.load.atlas()`. This becomes critical before you add autotile variants.

### Q45: Sprite generation quality control -- SDXL produces sprite sheets and opaque backgrounds for certain prompts (the crystal icon needed 3 attempts). How do you automate rejection at scale?
**Context**: The MEMORY.md notes that "SDXL keeps producing sprite sheets" and the crystal icon required 3 regeneration attempts. At 500+ sprites needed for 25 biomes, a 30% rejection rate means ~150 wasted generations.
**My Advice**: Build a 3-gate automated QC pipeline between generation and human review: (1) Composition gate -- use a lightweight classifier or edge detection to reject images with more than one distinct object (catches the 4-up sprite sheet problem). (2) Background gate -- after rembg processing, reject images where the mean alpha in the outer 10% of pixels exceeds 30% (catches incomplete background removal). (3) Palette gate -- reject images whose dominant colors differ from the target biome palette by more than a configurable threshold. Run this pipeline automatically; only images that pass all 3 gates go to human review. Target: reduce rejection rate from ~30% to under 5%.

### Q46: The game renders at TILE_SIZE=32 in the mine and 4px tiles in the dome. When transitioning from dome to mine (clicking "Dive"), the visual scale changes dramatically. How do you smooth this transition?
**Context**: The dome renders at 4px/tile in a 768x512 canvas (CSS scaled up). The mine renders at 32px/tile in Phaser. The player perceives completely different zoom levels and art styles across these two core views.
**My Advice**: Design a transition animation that bridges the scale gap. When the player clicks the dive hatch: (1) camera zooms into the dive hatch over 0.5s, (2) cross-fade from the dome canvas to a dark tunnel backdrop over 0.3s, (3) the dark tunnel transitions into the mine's first visible frame. The zoom-in creates a visual "portal" effect that makes the scale change feel intentional rather than jarring. On surfacing, reverse the sequence: zoom out from the mine spawn point, cross-fade through the tunnel, arrive at the dome at full zoom.

### Q47: Mobile draw call budget -- Phaser uses WebGL. How many draw calls can you sustain at 60fps on a 2020 mid-range Android phone?
**Context**: The current `drawTiles()` loop renders each visible tile individually via `getPooledSprite()`, which creates/reuses Phaser Image objects. Each Image with a different texture key is a separate draw call. At 12x15 = 180 visible tiles, that is potentially 180 draw calls per frame, plus overlays, particles, and the player sprite.
**My Advice**: Budget 200-250 draw calls per frame for a 2020 mid-range phone at 60fps. You are currently near this limit with 180 tile sprites + overlay graphics + particles. When autotiling multiplies the number of unique texture keys, you will blow past this budget. The solution: pack sprites into texture atlases (per biome) so that all tiles in the same biome share one atlas texture -- Phaser can then batch them into far fewer draw calls. With atlased sprites, 180 tiles from the same biome become ~1-3 draw calls.

### Q48: Resolution scaling strategy -- 32px sprites are tiny on high-DPI phones (3x). Do you render at native device resolution or at a fixed logical resolution?
**Context**: The mine uses `TILE_SIZE: 32` from balance.ts. On an iPhone 15 at 3x DPI, each "32px" tile is actually 96 physical pixels. The game currently does not adjust for device pixel ratio. Phaser defaults to the CSS pixel size of the canvas, not the device pixel ratio.
**My Advice**: Render the Phaser canvas at 1x CSS pixel resolution, not at native device DPI. At 3x DPI, rendering every tile at 96x96 physical pixels would triple your GPU fill rate for no visual benefit (pixel art should not be anti-aliased). Let CSS `image-rendering: pixelated` handle the upscale to physical pixels. This keeps GPU cost identical across all devices. If you later want sharper rendering on tablets, switch to the 256px hi-res sprite set and render at 2x -- but never render at native DPI for pixel art.

### Q49: You have 65 sprites in the low-res set and 63 in the hi-res set (based on the current asset counts). At 500+ sprites for 25 biomes, what is your estimated total decoded texture memory, and does it fit in a mobile GPU memory budget?
**Context**: Each 256px PNG, decoded to RGBA, is 256KB of GPU texture memory. At 500 individual 256px sprites, that is ~125MB of decoded texture memory. Mobile GPUs typically have 1-2GB of shared memory, but other apps and the OS consume most of it.
**My Advice**: Budget 80-100MB max for decoded textures on mobile. At 256px per sprite, that is roughly 320-400 individual sprites. You will exceed this with 500+ biome sprites at 256px. Solutions: (1) Use the 32px set as the default mobile resolution (32px RGBA = 4KB per sprite, 500 sprites = only 2MB). (2) Load only the current biome's atlas, plus shared UI sprites. Unload the previous biome's atlas on layer transition. (3) Generate a 64px intermediate set (16KB per sprite, 500 sprites = 8MB) for a quality/memory sweet spot on mobile. Never load all 25 biomes' sprites simultaneously.

### Q50: Build-time asset audit -- as you scale to 500+ sprites, how do you prevent orphan sprites (generated but never wired into code)?
**Context**: BUG-006 documents 5 existing orphan sprites (`mineral_blue.png`, `mineral_green.png`, `mineral_red.png`, `relic_gold.png`, `relic_tablet.png`) that are committed but unreferenced. At 500+ sprites, manual tracking of which assets are live becomes impossible.
**My Advice**: Implement a build-time asset audit script that runs as part of `npm run build`. The script scans all files in `src/assets/sprites/` and `src/assets/sprites-hires/`, then searches the codebase (spriteManifest.ts, domeManifest.ts, MineScene.ts, and all Svelte components) for references to each sprite key. Unreferenced sprite files emit a build warning. Code references to missing sprite files emit a build error. This turns orphan sprites from hidden technical debt into a visible, catchable issue. At 500+ assets, you cannot rely on human memory -- automate the audit.
