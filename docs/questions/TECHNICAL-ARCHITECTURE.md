## A. Phaser Engine & Rendering Performance (Questions 1-10)

### Q1: Should MineScene migrate from Graphics-based tile rendering to a Phaser Tilemap layer before autotiling lands?
**Context**: `MineScene.drawTiles()` currently iterates every visible cell each frame, calling `tileGraphics.clear()` and re-drawing via `drawBlockPattern()` with pooled `Phaser.GameObjects.Image` sprites. The `update()` method calls `redrawAll()` unconditionally. With Phase 8 scaling grids to 40x40 (1,600 cells) and Phase 7 adding 16-47 autotile variants per block type, the per-frame full-redraw cost will multiply significantly.
**My Advice**: Migrate to `Phaser.Tilemaps.TilemapLayer` with pre-computed tile indices stored in each `MineCell`. Phaser's tilemap renderer batches draw calls into a single GPU call per layer, reducing your current O(visible_cells) individual `setTexture`/`setPosition` calls to O(1) per tilemap layer. Pre-compute the autotile bitmask index whenever a cell changes (mined, revealed, earthquake collapse), not every frame. This is the single highest-impact rendering optimization available to you.

### Q2: What is the draw call budget per frame, and how do you enforce it across mine, overlay, particle, and UI layers?
**Context**: MineScene currently uses at least 5 rendering layers: `tileGraphics` (depth 0), pooled item sprites (depth 5), `overlayGraphics` (depth 7), `playerSprite` (depth 10), and `breakEmitter` particles. Phase 7 adds per-block idle animations, ambient biome particles, loot physics sprites, and character animation frames. Each layer generates separate draw calls, and the overlay graphics layer draws dozens of individual fillRect/strokeRect/fillCircle calls for fog, scanner hints, and hazard warnings.
**My Advice**: Target a hard cap of 50 draw calls per frame for mobile (mid-range Android WebView). Consolidate all static tile rendering into a single tilemap atlas draw, all fog/overlay effects into a second RenderTexture that is updated only on visibility changes, and reserve dynamic draw calls exclusively for particles, player animation, and loot physics. Profile on a real Pixel 6a or Samsung A-series device -- Chrome DevTools throttling dramatically understates actual mobile GPU pressure.

### Q3: The current sprite pool (`itemSpritePool`) creates new `Phaser.GameObjects.Image` objects on demand and never shrinks. At 40x40 grids with biome-specific block idle animations, how large could this pool grow and what is the memory ceiling?
**Context**: `getPooledSprite()` in MineScene pushes new images onto `itemSpritePool` when the pool index exceeds current length, but the pool is never trimmed. After visiting a large mine layer with many visible cells, the pool could hold hundreds of Image objects. With Phase 7 adding animated blocks (crystal shimmering, lava bubbling, artifact glowing) requiring multiple sprites per cell, the pool could exceed 1,000 objects.
**My Advice**: Implement a hard ceiling on the sprite pool (e.g., 500 objects) and switch to frustum culling -- only allocate sprites for cells within the camera viewport plus a 2-tile margin. When the player scrolls, recycle sprites from cells that left the viewport rather than creating new ones. This keeps memory bounded regardless of grid size and is essential for the 20-layer expedition model where each layer allocates a fresh pool.

### Q4: Should block idle animations (Phase 7.7 -- crystal glinting, lava flowing, gas swirling) be implemented as Phaser animation frames on individual sprites, or as shader effects on a shared tilemap?
**Context**: Phase 7.7 specifies 8+ block types with idle animations (2-6 frames each). If each animated block is a separate `Phaser.GameObjects.Sprite` with its own animation state machine, a layer with 50 animated blocks visible simultaneously means 50 independent animation timers and 50 texture swaps per animation tick. Phaser's animation manager handles this, but the per-object overhead adds up on mobile.
**My Advice**: Use a hybrid approach. For the tilemap layer (terrain blocks like lava, gas), animate by cycling the tile index on the tilemap at fixed intervals (e.g., every 200ms swap all lava tiles to the next frame simultaneously). This is a single tilemap update, not per-sprite. For overlay-layer objects (artifact glow, crystal sparkle), use Phaser's sprite-based animations since these are fewer in number and need individual timing. Never mix per-frame Graphics-based procedural animation (like the current lava glow dot in `drawBlockPattern`) with sprite-based animation -- pick one system.

### Q5: Phase 7.4 specifies ambient biome particles (embers in volcanic, sparkles in crystalline, spores in fungal). With 25 biomes each having unique particles active continuously, what is the particle budget?
**Context**: MineScene currently has a single `breakEmitter` (6 particles per burst, 400ms lifespan). Phase 7.4 adds 8+ ambient particle types plus block-break particles, oxygen leak particles, and achievement bursts. Phaser 3's particle system creates lightweight pooled particles, but each emitter still occupies a scene game object slot, and each active particle requires per-frame position/alpha/scale interpolation.
**My Advice**: Cap total active particles at 200 for mobile, 500 for desktop. Use two emitter managers: one "ambient" emitter per biome (reused across blocks, positioned at camera center, particles drift randomly within viewport) and one "event" emitter pool for bursts (block break, loot pop, achievement). The ambient emitter should run at reduced frequency when battery saver mode is detected via `navigator.getBattery()`. Budget approximately 3-5% of frame time for particle updates -- measure this on a real device.

### Q6: The camera currently uses a manual lerp in `updateCameraTarget()` with a fixed lerp factor of 0.15. Phase 7.3 requires smooth follow, pinch-to-zoom, drag-to-pan, and a mini-map. Should you replace the manual camera with Phaser's built-in camera follow and zoom systems?
**Context**: `updateCameraTarget()` manually computes `cam.scrollX += (desiredX - cam.scrollX) * lerp`. Phaser 3 has `camera.startFollow(target, true, lerpX, lerpY)` with built-in deadzone support, `camera.setZoom()`, and `camera.pan()` -- all of which handle bounds clamping, smooth interpolation, and input gesture integration. The manual approach currently duplicates Phaser's native functionality while missing features like deadzone and zoom.
**My Advice**: Switch to Phaser's native camera follow system and remove the manual lerp code entirely. Use `camera.startFollow(cameraTarget, true, 0.15, 0.15)` and configure a deadzone so the camera only moves when the player nears the viewport edge. For pinch-to-zoom, use Phaser's `input.on('wheel')` for desktop and add a Hammer.js or native `gesturechange` listener for mobile pinch. The mini-map can be a second Phaser camera rendering the same scene at 0.1x zoom with a fixed position overlay.

### Q7: Phase 10 envisions the dome as a multi-floor hub with animated pets, interior lighting, and ambient animations. The current DomeCanvas uses a Svelte-managed offscreen canvas outside of Phaser. Should the dome migrate to a dedicated Phaser scene for hardware-accelerated rendering?
**Context**: `DomeCanvas.svelte` renders tiles on an offscreen canvas with manual `drawImage()` calls, CSS-scaled to display size. This works for the current static dome but cannot efficiently support: per-pet pathfinding and walk animations, dynamic interior lighting that responds to upgrades, animated machinery per room, or floor transition animations. Phaser's multi-scene support allows running a dome scene alongside the mine scene with shared asset caches.
**My Advice**: Migrate the dome to a dedicated Phaser scene (`DomeScene`). This gives you sprite batching, animation state machines, tweens for floor transitions, and physics for pet movement -- all of which you would otherwise re-implement poorly in raw Canvas2D. Use Phaser's scene manager to switch between `DomeScene` and `MineScene`, sharing the same Phaser Game instance. The Svelte overlay layer (resource bars, panels, popups) remains as DOM overlays on top of the Phaser canvas, exactly as it does now for the mine.

### Q8: With 25 biomes each requiring unique tile sprite sheets (Phase 9), plus autotile variants, the total texture memory could reach hundreds of megabytes. What is your texture atlas strategy and loading architecture?
**Context**: Currently `spriteManifest.ts` uses `import.meta.glob` to eagerly load all sprites at boot. At the current scale (~50 sprites at 32px or 256px) this is fine. Phase 9 adds 125+ base tile variants, each potentially needing 16-47 autotile sub-tiles. If all 256px variants are loaded eagerly, total decoded texture memory could easily exceed 500MB, crashing mobile WebViews.
**My Advice**: Implement per-biome texture atlases loaded on demand. Pack all tiles for a single biome into one atlas PNG with a JSON coordinate map (Phaser has native `atlas` support via `this.load.atlas()`). On layer transition, load the next biome's atlas during the descent animation, then unload the previous biome's atlas after the transition completes. Keep a maximum of 3 biome atlases in memory simultaneously (current + next + previous for smooth back-tracking). Use TexturePacker or a Sharp-based build script to generate atlases at build time.

### Q9: Phaser's `GameConfig` is set to `Phaser.AUTO` which selects WebGL when available, falling back to Canvas2D. Have you verified that all rendering code paths (Graphics draws, overlays, blend modes) work identically in both renderers?
**Context**: `GameManager.boot()` passes `type: Phaser.AUTO` to the Phaser config. The mine scene uses `Phaser.GameObjects.Graphics` extensively for tile backgrounds, fog overlays, crack lines, scanner hints, and hazard indicators with alpha blending, fill styles, and stroke styles. Some Graphics operations behave differently between Canvas and WebGL renderers -- particularly alpha compositing, lineStyle rendering at sub-pixel coordinates, and blend modes.
**My Advice**: Force `Phaser.WEBGL` in the config and drop Canvas2D support entirely. Every modern mobile browser and Capacitor WebView supports WebGL. By standardizing on one renderer, you avoid subtle cross-renderer bugs, gain access to WebGL-only features (custom shaders for fog of war, post-processing effects for screen shake), and simplify testing. If you ever encounter a device without WebGL, a "device not supported" message is better than a broken rendering fallback.

### Q10: The mine generation in `generateMine()` runs synchronously on the main thread. At 40x40 grids with micro-structures, hazard placement, and flood-fill BFS for visibility, how long does generation take on mobile and should it be moved to a Web Worker?
**Context**: `generateMine()` builds the full grid, places special blocks, seeds micro-structures (Ancient Library 7x5, Ruins 9x7, etc.), runs `placeSpecialBlocks()` multiple times, and performs various depth-based calculations. The current 20x40 grid (800 cells) generates near-instantly, but Phase 8 scales to 40x40 (1,600 cells) with more complex structure placement. Additionally, `revealAround()` uses flood-fill BFS through empty cells, which at spawn could touch hundreds of cells.
**My Advice**: Profile generation time on a mid-range phone at 40x40 first -- it is likely still under 50ms, which is acceptable for a synchronous call during a layer transition (the descent animation masks it). Only move to a Web Worker if profiling shows >100ms on target devices. If you do need a Worker, the seeded RNG (`mulberry32`) and pure-function generation code are already perfectly Worker-compatible since they have no DOM dependencies. The bigger concern is the `revealAround()` BFS on large grids with many empty pockets -- cap BFS depth at 200 cells to prevent worst-case explosions.

## B. State Management, Save System & Offline Architecture (Questions 11-20)

### Q11: PlayerSave is a single monolithic JSON blob with 30+ top-level fields. As Phases 12-16 add interest preferences, behavioral tracking, guild membership, and crop farming state, should you split it into domain-specific sub-objects now?
**Context**: The `saves` table stores `save_data` as a single `text` column containing the full serialized `PlayerSave`. Every sync operation uploads and downloads the entire blob. The `resolveConflict()` method in `syncService.ts` uses a simple `lastPlayedAt` timestamp comparison, which means whichever device saves last wins entirely -- there is no field-level merge.
**My Advice**: Split `PlayerSave` into versioned sub-documents: `PlayerEconomy` (minerals, crafting counts), `PlayerKnowledge` (review states, learned facts, knowledge points), `PlayerCollection` (fossils, cosmetics, relics), `PlayerProgress` (stats, streaks, milestones), `PlayerPreferences` (interests, GAIA settings). Each sub-document gets its own `lastModifiedAt` timestamp. Conflict resolution becomes per-subdocument (economy conflicts resolve via max, knowledge merges via latest review date). This also enables partial sync -- upload only the changed sub-document, reducing bandwidth by 80-90% for typical saves.

### Q12: The save system uses `localStorage` exclusively. A fully-progressed player with 5,000+ review states, complete fossil/cosmetic/relic collections, and behavioral preference data could exceed localStorage's 5-10MB limit. Have you measured projected save size at endgame scale?
**Context**: `saveService.ts` calls `localStorage.setItem('terra_save', JSON.stringify(save))`. The SM-2 review state for each fact includes quality, easeFactor, interval, repetitions, and lastReviewAt. At 5,000 facts with full review state, that alone is approximately 500KB of JSON. Add minerals (5 tiers), 50+ fossil states, crafting counts, purchased items, farm slots, behavioral preference vectors, and the total could approach 2-3MB.
**My Advice**: Measure immediately by creating a synthetic endgame save and serializing it. If it exceeds 1MB, migrate persistence to IndexedDB using the `idb-keyval` library (600 bytes, localStorage-like API, effectively unlimited storage). Keep localStorage as a fallback for the save version number only. IndexedDB also supports structured cloning, which is faster than JSON.stringify/parse for large objects. Do this before Phase 11 adds thousands more facts.

### Q13: The sync conflict resolution (`resolveConflict` in syncService.ts) is last-write-wins based on `lastPlayedAt`. If a player studies 50 facts on their tablet (offline), then plays a mine run on their phone (online, syncs first), the tablet's 50 review state updates are silently lost. Is this acceptable?
**Context**: `resolveConflict()` returns whichever save has the higher `lastPlayedAt`. There is no merge logic -- the entire save is either local or remote. The debounce timer (30 seconds) means rapid-fire saves on one device will batch into a single upload, but there is no mechanism to detect or resolve divergent edits across devices.
**My Advice**: For a learning app, losing review state data is catastrophic -- it undermines the entire SM-2 spaced repetition system. Implement field-level merge: review states merge by taking the entry with the latest `lastReviewAt` per fact ID; mineral counts take the maximum; stats take the maximum; inventory unions with deduplication. The merge function should be a pure function that can be unit-tested exhaustively. Document the merge semantics clearly since they affect game balance (max-merge on minerals is slightly exploitable but vastly preferable to data loss).

### Q14: The `apiClient.ts` stores JWT access and refresh tokens in `localStorage`, which is accessible to any JavaScript running on the same origin. For a Capacitor native app where XSS risk is lower, is this acceptable, or should tokens move to httpOnly cookies or Capacitor Secure Storage?
**Context**: `apiClient` reads/writes `terra_auth_token` and `terra_refresh_token` to localStorage. The CSP in `index.html` restricts script-src to 'self', and Capacitor's WebView restricts navigation to the app origin. However, any XSS vulnerability (e.g., via a crafted fact distractor containing script content that bypasses sanitization) could exfiltrate these tokens.
**My Advice**: For the web version, migrate refresh tokens to `httpOnly` + `Secure` + `SameSite=Strict` cookies managed by the Fastify backend. The access token can remain in-memory (not localStorage) with a 15-minute expiry, refreshed via the cookie-based endpoint. For the Capacitor native app, use `@capacitor/preferences` (formerly Storage) which uses iOS Keychain / Android EncryptedSharedPreferences. This layered approach gives you best-available security per platform without breaking the offline-first model.

### Q15: There is no Service Worker implementation despite the architecture document mentioning "Service Worker caches all static assets." When should this be built, and what caching strategy should it use?
**Context**: `docs/ARCHITECTURE.md` describes a Service Worker for static asset caching and stale-while-revalidate for quiz data, but no `sw.js` or service worker registration exists in the codebase. Capacitor's WebView on Android handles offline differently from a PWA -- assets are served from the local filesystem, not a web server.
**My Advice**: For the Capacitor native app, a Service Worker is unnecessary -- assets are already bundled in the APK. For the web version (dev/preview), implement a Service Worker using `vite-plugin-pwa` with: (1) precache for all built assets in `dist/`, (2) stale-while-revalidate for `facts.db`, (3) network-first for API endpoints with fallback to cached responses, (4) offline fallback page. Build this in Phase 19 alongside the auth system, not before -- it adds complexity to development with cache invalidation headaches that slow down iteration.

### Q16: The `factsDB` loads the entire `facts.db` WASM database into memory on init. At 5,000+ facts with 25 distractors each plus visual descriptions, what is the projected file size and mobile load time?
**Context**: `scripts/build-facts-db.mjs` builds `public/facts.db` from JSON seed files. `factsDB.init()` fetches this file and initializes sql.js with the buffer. Currently at 522 facts, the file is small. Phase 11 targets thousands of facts with expanded metadata fields (visual_description, image_prompt, hierarchical categories, 25 distractors each with multiple fields).
**My Advice**: Estimate: 5,000 facts x ~2-3KB each (including 25 distractors at ~50 chars each) = 10-15MB uncompressed. With gzip, this drops to ~3-5MB over the network. On a 4G connection, that is 1-3 seconds of load time. For the Capacitor app, bundle a baseline `facts.db` in the APK (`public/` directory ships with the build) and implement delta sync from the server for new/updated facts. For the web version, implement incremental loading: fetch a minimal `facts-core.db` (question + answer only, ~2MB) at boot, then lazy-fetch full metadata (distractors, explanations) when a quiz is triggered.

### Q17: When the fact pipeline (Phase 11) adds facts via the server API, how do those new facts reach the client's bundled WASM database?
**Context**: Currently facts are baked into `public/facts.db` at build time from `src/data/seed/*.json` files. There is no runtime mechanism to add, update, or remove facts. The Capacitor native app would require an app store update to ship new facts, which defeats the "weekly fact batches" cadence planned in Phase 23.
**My Advice**: Implement a fact sync protocol: (1) Server exposes `GET /api/facts/delta?since={version}` returning facts added/modified since the client's version, (2) Client stores a `factDbVersion` number in PlayerSave, (3) On app foreground or post-dive, check for deltas in the background, (4) Apply deltas to the local sql.js database using INSERT OR REPLACE, (5) Persist the updated database to IndexedDB so it survives page reloads. This decouples content delivery from app releases entirely. The APK-bundled `facts.db` serves as the offline fallback for first launch.

### Q18: The sync debounce in `syncService.ts` is 30 seconds. If the app is killed (mobile task switch, battery death, crash) within that window, the latest local save never reaches the cloud. How do you prevent data loss?
**Context**: `syncAfterSave()` checks `msSinceLastAttempt < SYNC_DEBOUNCE_MS` and schedules a deferred upload via `setTimeout`. If the process is terminated before the timer fires, the upload is lost. On mobile, apps are routinely killed by the OS to reclaim memory, often without `beforeunload` or `visibilitychange` events firing.
**My Advice**: Use a two-pronged approach: (1) Register a `visibilitychange` listener and Capacitor's `App.addListener('appStateChange', ...)` to trigger an immediate sync (bypass debounce) when the app goes to background, (2) Write every save to IndexedDB synchronously as the primary persistence layer (fast, durable, no size limits), and treat cloud sync as an eventually-consistent backup. The debounce should only throttle cloud uploads, not local persistence. This ensures zero data loss even on abrupt termination.

### Q19: The current app has no error boundary for API failures. If the backend is down during a sync attempt, `console.warn` is the only indicator. Should there be user-visible sync status feedback?
**Context**: Both `syncService.ts` and `apiClient.ts` swallow errors with `console.warn`. The `SyncService` exposes `isSyncing` and `lastSyncTime` properties but no component reads them. A player could play for days believing their progress is backed up when the server has been unreachable the entire time.
**My Advice**: Add a small sync indicator to the dome UI -- a cloud icon that is green (synced within last hour), yellow (sync pending/in-progress), or red (last sync failed or >24 hours ago). Tapping it shows the last successful sync timestamp and a manual "Sync Now" button. For critical sync failures (3+ consecutive failures), show a non-blocking GAIA toast: "My databanks are having trouble connecting to the satellite. Your progress is safe locally." Never show technical error messages -- always through GAIA's personality.

### Q20: DD-V2-043 mandates offline-first architecture. But many planned features (leaderboards, knowledge duels, hub visiting, daily deals seeded by server time, fact sync) require connectivity. How do you gracefully degrade each feature when offline?
**Context**: The roadmap includes Phase 22 (social/multiplayer), Phase 23 (live ops/seasons), and Phase 19 (cloud sync) -- all server-dependent. Meanwhile, DD-V2-043 says "fully locally playable, cloud sync is a backup." The daily deals system (`dailyDeals.ts`) already uses `new Date()` for seeding, which works offline, but leaderboards, duels, and guild features fundamentally cannot.
**My Advice**: Define three tiers of feature availability: (1) **Always available offline**: mining, quizzes, study sessions, Knowledge Tree, crafting, farm, dome navigation, fossil revival -- the core loop never requires connectivity. (2) **Cached when available**: leaderboards (show last-fetched snapshot), daily deals (client-seeded, server validates on sync), GAIA journey memory (local templates only). (3) **Online-only**: knowledge duels, hub visiting, guild challenges, fact trading, IAP. Display online-only features with a "Requires connection" badge when offline rather than hiding them, so players know what they are missing.

## C. Backend Architecture & Database Design (Questions 21-30)

### Q21: The backend uses SQLite via `better-sqlite3` for development but plans PostgreSQL for production. The `saves` table stores the entire PlayerSave as a JSON text blob. Should you use PostgreSQL's `jsonb` type instead, and should frequently-queried fields be extracted into proper columns?
**Context**: `server/src/db/schema.ts` defines `saveData: text("save_data")` containing the full serialized PlayerSave. Analytics queries (find users with streaks > 30, segment by total facts learned, identify churning players) would require deserializing every save row. Leaderboard scores are submitted separately but could be derived from the save data if it were queryable.
**My Advice**: Use `jsonb` for the save blob (enables `save_data->>'stats'->>'totalDivesCompleted'` queries without deserialization) AND extract hot analytics fields into dedicated columns: `last_played_at`, `total_dives`, `total_facts_learned`, `current_streak`, `account_tier`. This gives you fast analytics queries without restructuring the entire save format. The denormalized columns are updated atomically with the save blob on each upload.

### Q22: The `saves` table allows multiple rows per user as "a history." At a 30-second sync debounce, an active player generates ~120 save rows per hour. What is your retention policy, and how do you query the latest save efficiently?
**Context**: There is no `is_active` flag or index to identify the current save. Fetching the latest save requires `ORDER BY updated_at DESC LIMIT 1` with a full index scan. The saves route presumably does this, but at scale with thousands of users each having hundreds of save history rows, this query becomes expensive.
**My Advice**: Add an `is_active` boolean column with a partial unique index: `CREATE UNIQUE INDEX ON saves(user_id) WHERE is_active = TRUE`. On upload, atomically set the old active row to `is_active = false` and insert the new row as `is_active = true` in a transaction. Implement a retention policy: keep the 10 most recent saves per user, prune older rows via a scheduled job (Fastify `@fastify/schedule` or a cron endpoint). This bounds storage growth and makes latest-save queries O(1) via the unique index.

### Q23: The leaderboard system stores one row per submission event and computes rankings on read. With 10K users, 5+ categories, and frequent submissions, how does this scale?
**Context**: `leaderboards` table has no composite index on `(category, score)`. Each `getLeaderboard()` call likely scans all rows for a category and sorts by score. The `submitScore` endpoint inserts a new row on every sync (via `submitLeaderboardScores` in syncService), even if the score has not changed.
**My Advice**: Three changes: (1) Add a composite index `CREATE INDEX ON leaderboards(category, score DESC)` for efficient top-N queries. (2) Change the submission model to upsert -- store only the best score per (userId, category) pair with a unique constraint, not every submission event. This reduces table size by orders of magnitude. (3) Cache the top-100 per category in memory (Fastify `onReady` hook + 60-second refresh timer) since leaderboard reads vastly outnumber writes. Consider a separate `leaderboard_history` table if you need score progression tracking.

### Q24: The auth system uses PBKDF2-SHA512 with what appears to be synchronous hashing. At scale, how does this affect request throughput, and should you switch to Argon2id?
**Context**: `server/src/routes/auth.ts` uses `config.pbkdf2Iterations` (presumably 100K+) for password hashing. Node.js `crypto.pbkdf2Sync` blocks the event loop during computation. At 100K iterations, a single hash takes ~50-100ms. During a signup/login surge, even 10 concurrent auth requests would block the event loop for 500ms-1s, delaying all other requests (health checks, save uploads, leaderboard queries).
**My Advice**: Immediately switch to the async `crypto.pbkdf2()` version (callback-based, runs on libuv thread pool). Better yet, migrate to Argon2id via the `argon2` npm package -- it is memory-hard (resistant to GPU brute-force), natively async, and the OWASP-recommended algorithm for 2025+. Use `argon2.hash(password, { type: argon2.argon2id, memoryCost: 19456, timeCost: 2, parallelism: 1 })` which takes ~100ms but does NOT block the event loop. Implement a transparent migration: on login, if the stored hash is PBKDF2, verify it, then re-hash with Argon2id and update the row.

### Q25: DD-V2-042 specifies Google and Apple social login. How does this integrate with the current email/password JWT system, and what is the account linking flow for guest-to-authenticated?
**Context**: The current auth system only supports email/password. Social login requires OAuth2 flows with provider-specific libraries, token exchange, and identity linking. Guest mode requires generating an anonymous account that can later be upgraded without losing progress. The Capacitor native app needs platform-specific sign-in SDKs (Google Sign-In for Android, Sign in with Apple for iOS).
**My Advice**: Use Capacitor plugins `@nicegoodthings/capacitor-google-auth` and `@nicegoodthings/capacitor-sign-in-with-apple` for native OAuth flows. On the backend, add a `POST /api/auth/social` endpoint that accepts a provider token, verifies it with the provider's API, and either finds an existing linked user or creates one. For guest accounts: generate a UUID on first launch, create a "guest" user row (no email, no password), issue a JWT. Account linking (`POST /api/auth/link`) associates the guest user's save data with the newly authenticated social/email identity. Store `auth_provider` and `provider_id` columns on the users table.

### Q26: The fact ingestion API (Phase 11) will process facts through LLM pipelines for distractor generation, categorization, and metadata enrichment. Where does the LLM run, what model, and what is the cost model?
**Context**: DD-V2-001 and DD-V2-002 specify fully LLM-generated content fields. Phase 11.3 lists 10+ auto-generated fields per fact (distractors, gaiaComment, explanation, wowFactor, quiz_question, correct_answer, fun_score, age_rating, image_prompt, mnemonic). At 5,000+ facts, this is a significant batch processing job.
**My Advice**: Use a cloud LLM API (Claude Sonnet or GPT-4o-mini) for the content pipeline -- quality on educational content justifies the cost, and local models lack the nuance needed for plausible distractor generation. Estimate: ~1,500 tokens per fact (input prompt + all generated fields) at ~$0.003/1K tokens = ~$0.005/fact. 5,000 facts = ~$25. This is negligible. Run the pipeline as a server-side batch job with a queue table (`facts_processing_queue`), rate-limited at 10 requests/second to stay within API limits. Store raw LLM responses alongside parsed fields for debugging and regeneration.

### Q27: Phase 11.1 requires semantic duplicate detection for incoming facts. What similarity approach provides the best accuracy-to-complexity tradeoff?
**Context**: When a new fact is submitted to the ingestion API, it must be checked against the existing corpus for duplicates. Simple string matching misses semantic duplicates ("The heart has 4 chambers" vs "A human heart contains four chambers"). The fact corpus will grow from 522 to 5,000+ entries.
**My Advice**: Use text embeddings with cosine similarity. Generate embeddings for all facts using `text-embedding-3-small` (OpenAI, $0.02/1M tokens -- effectively free at this scale) or a local sentence-transformers model. Store embeddings in PostgreSQL using the `pgvector` extension (supported on Railway, Fly.io, Supabase). On ingestion, compute the incoming fact's embedding and query `SELECT * FROM facts ORDER BY embedding <=> $1 LIMIT 5` -- any result above a 0.85 cosine similarity threshold is flagged as a potential duplicate. This is more accurate than TF-IDF and scales to 100K+ facts with a HNSW index.

### Q28: The production database migration from SQLite to PostgreSQL (Phase 19.3) is a critical operation. What is your migration strategy that ensures zero data loss?
**Context**: The server currently uses `better-sqlite3` with Drizzle ORM. The schema uses SQLite-compatible types (`text`, `integer`) that map to PostgreSQL, but behavioral differences exist (SQLite's dynamic typing, implicit rowid, AUTOINCREMENT semantics). Drizzle supports dialect switching, but the migration itself (transferring existing user data) needs careful planning.
**My Advice**: Phase the migration: (1) Write a Drizzle schema that targets PostgreSQL (`drizzle-orm/pg-core`). Run `drizzle-kit generate:pg` to produce the DDL. (2) Deploy the PostgreSQL instance alongside SQLite. (3) Write a data migration script that reads all SQLite rows and inserts them into PostgreSQL with data transformation (e.g., `integer` timestamps to proper `timestamp` columns if desired). (4) Run both databases in parallel for 1 week (dual-write mode). (5) Switch reads to PostgreSQL, verify, remove SQLite. Critically: the migration script must be idempotent and re-runnable, and the dual-write period catches any dialect-specific bugs before they affect users.

### Q29: The `refreshTokens` table stores raw token strings as primary keys. Should refresh tokens be hashed before storage, and what is the rotation/revocation strategy?
**Context**: `refresh_tokens.token` is stored as plaintext. If the database is compromised (SQL injection, backup leak, admin access), an attacker gains valid refresh tokens for all users. The current rotation strategy (delete old token, issue new one) is correct but offers no revocation mechanism beyond deleting the row.
**My Advice**: Hash refresh tokens with SHA-256 before storage (fast, one-way, no rainbow table risk for high-entropy tokens). Store the hash as the primary key; the client holds the raw token. On refresh, hash the incoming token and look up the hash. For revocation: add a `family_id` column (UUID assigned at login, shared across rotated tokens). If a refresh token is used after it has been rotated (replay attack detection), revoke ALL tokens in that family -- this is the "refresh token rotation with reuse detection" pattern recommended by OAuth 2.0 Security BCP.

### Q30: Rate limiting is mentioned in `docs/SECURITY.md` but not implemented. What rate limiting strategy should protect auth endpoints, save uploads, leaderboard submissions, and the fact ingestion API?
**Context**: The Fastify server has no rate limiting. Auth endpoints (login, register, refresh) are vulnerable to brute-force attacks. Save upload with its 30-second debounce is self-limiting on the client side but has no server-side enforcement. The fact ingestion API (Phase 11) could be abused to trigger expensive LLM calls.
**My Advice**: Use `@fastify/rate-limit` with per-route configuration: auth endpoints at 5 requests/minute per IP (prevents brute-force), save upload at 2 requests/minute per user (matches client debounce), leaderboard submit at 10/minute per user, fact ingestion at 10/minute per admin user. For the LLM pipeline, add a separate cost-based rate limit: track daily LLM API spend in a Redis counter and circuit-break at a configurable threshold (e.g., $5/day). Return `429 Too Many Requests` with a `Retry-After` header for all rate limit violations.

### Q36: The mine generation uses `seededRandom()` (mulberry32 algorithm) for deterministic procedural generation. How do you ensure that code changes never accidentally break seed determinism?

## E. Architecture Patterns, Security & Scaling (Questions 41-50)

### Q41: The GameManager is a singleton that bridges Phaser and Svelte, manages dive lifecycle, and delegates to 4 sub-managers. As Phases 10-16 add dome navigation, pet AI, farm timers, and behavioral tracking, should GameManager be further decomposed or replaced with an event-driven architecture?
**Context**: `GameManager.ts` is 1,217 lines after decomposition (down from 1,544). It still owns: game boot, dive start/end, layer transitions, mine scene event handling, companion effects, biome sequencing, and coordinates the Quiz/Study/GAIA/Inventory sub-managers. Adding dome miner pathfinding, pet behavior, farm harvest timers, and interest tracking would push it back toward monolithic.
**My Advice**: Extract a `DiveManager` (dive lifecycle, layer transitions, biome sequence) and a `DomeManager` (floor navigation, pet AI, farm timers, GAIA idle behavior) from GameManager. GameManager becomes a thin coordinator that owns the Phaser Game instance and routes between DiveManager and DomeManager based on the current screen. Each manager subscribes to a shared typed event bus rather than calling each other directly. This keeps each manager under 500 lines and makes adding new systems (behavioral tracking, social features) a matter of adding new subscribers, not modifying existing managers.

### Q42: Should the project adopt an Entity Component System (ECS) pattern for the mine scene, or is the current object-oriented approach sufficient for the planned complexity?
**Context**: The mine currently uses a simple `MineCell[][]` grid with `Player` entity and procedural rendering. Phase 7 adds per-block animations, Phase 8 adds 20 layers, Phase 10 adds pet AI, and Phase 25 adds underground creatures and combat. An ECS would separate data (components) from behavior (systems), enabling better cache locality and easier addition of new entity types.
**My Advice**: Do not adopt a full ECS framework. The mine scene's entities are fundamentally grid-based, not free-moving -- ECS shines when you have many heterogeneous entities with different component compositions moving freely in space. Your grid cells are uniform structs, not entities. Instead, enrich the `MineCell` type with optional component-like fields (animation state, particle config, hazard timer) and add dedicated update systems that iterate the grid for their respective concerns (animation system, hazard system, visibility system). This gives you the separation-of-concerns benefit of ECS without the framework overhead.

### Q43: The Svelte-Phaser event bridge currently uses Phaser's `game.events.emit()` for Phaser-to-Svelte communication and direct method calls on GameManager for Svelte-to-Phaser. Should this be unified into a single typed event bus?
**Context**: When a block is mined, MineScene emits events via `this.game.events.emit('block-mined', data)`. GameManager subscribes with `this.game.events.on('block-mined', ...)`. For the reverse direction, Svelte components call `GameManager.getInstance().handleQuizAnswer(...)` directly. This asymmetry means adding a new cross-boundary interaction requires wiring in two different patterns.
**My Advice**: Introduce a singleton `EventBus` class with generic TypeScript support: `eventBus.emit<BlockMinedEvent>('block-mined', data)` and `eventBus.on<BlockMinedEvent>('block-mined', handler)`. Both Phaser scenes and Svelte components use the same bus. Define all event types in a central `src/events/types.ts` file. This decouples the systems completely -- a Svelte component can react to a Phaser event without knowing about GameManager, and a Phaser scene can react to a UI action without knowing about Svelte stores. The bus should be synchronous for game-critical events and async (microtask queue) for UI-update events to prevent Phaser frame drops.

### Q44: Client-side cheating prevention -- the game runs entirely in the client's browser/WebView. How do you validate claims (scores, mineral counts, fact mastery) before they reach the leaderboard?
**Context**: All game logic runs client-side. A player with browser DevTools can modify Svelte stores, call `saveService.save()` with fabricated data, and upload cheated saves to the cloud. Leaderboard scores are derived from save data in `syncService.submitLeaderboardScores()` with no server-side validation.
**My Advice**: Accept that client-side games cannot prevent all cheating, but add layered deterrents: (1) **Server-side plausibility checks**: reject saves where `totalFactsLearned > totalQuizzesTaken`, `deepestLayer > totalDives * MAX_LAYERS`, or mineral counts exceed theoretical maximums based on dive count. (2) **Rate anomaly detection**: flag accounts where progress velocity exceeds 3 standard deviations from the population mean. (3) **Leaderboard sanitization**: top-100 entries require manual approval or pass an automated plausibility check before display. (4) **Signed save checksums**: include a HMAC of critical save fields using a server-issued per-session secret, verify on upload. This does not prevent determined cheaters but stops casual DevTools editing.

### Q45: The CSP meta tag in `index.html` includes `'unsafe-inline'` for styles (required by Svelte) and `connect-src 'self' localhost:*` for development. What CSP changes are needed for production and Capacitor?
**Context**: `docs/SECURITY.md` notes that production CSP must replace localhost with the actual API domain. Capacitor on Android uses `https://` scheme (configured in `capacitor.config.ts`), meaning `connect-src 'self'` would only allow connections to the app's own scheme. The facts.db file, CDN assets, and API calls all need explicit CSP allowances.
**My Advice**: Create separate CSP headers for each environment: (1) **Development**: current permissive policy with `localhost:*`. (2) **Production web**: `default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data: blob: https://cdn.terragacha.com; connect-src 'self' https://api.terragacha.com; object-src 'none'; worker-src 'self' blob:` (add `blob:` for sql.js WASM worker). (3) **Capacitor**: use `@capacitor/http` for API calls (bypasses WebView CSP entirely for native HTTP) or add the API domain to Capacitor's `server.allowNavigation`. ### Q46: The `@fastify/jwt` configuration uses a single symmetric secret. Should you migrate to asymmetric signing (RS256/ES256) for production, and how does this affect token verification in a horizontally-scaled deployment?
**Context**: `server/src/index.ts` registers `@fastify/jwt` with `secret: config.jwtSecret`. In a multi-instance deployment behind a load balancer, every instance must share this secret. If one instance is compromised, the entire auth system is compromised.
**My Advice**: Migrate to RS256 (RSA) or ES256 (ECDSA) asymmetric signing. The private key lives in a secrets manager (Fly.io secrets, Railway env vars) and is used only for token signing. The public key can be distributed to any service that needs to verify tokens (future microservices, CDN edge functions, analytics pipeline) without exposing the signing capability. For horizontal scaling, only the auth-issuing instances need the private key; read-only instances use the public key. ES256 is preferred over RS256 for smaller tokens and faster verification.

### Q48: Phase 19.5 requires GDPR right to erasure (Article 17). Given that user data is spread across users, saves, leaderboards, refresh_tokens tables, plus client-side localStorage and IndexedDB, can you enumerate the complete deletion scope?
**Context**: A user deletion request must remove ALL personal data across all storage systems. The server has user data in 4 tables (with CASCADE deletes configured).
